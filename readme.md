# Software Requirements Specification (SRS)
## Система: «Кредитный навигатор»

## 1. Введение

### 1.1 Цель документа
Документ описывает текущие реализованные требования к веб-приложению «Кредитный навигатор» на основании фактической реализации в HTML/CSS/JavaScript.

### 1.2 Область применения
Система имитирует пользовательский путь подбора кредита:
- вход по телефону с SMS-кодом (мок);
- просмотр и выбор банковского предложения;
- заполнение анкеты;
- просмотр статуса заявки в профиле.

### 1.3 Технологический контекст
- Клиентская часть: HTML5, CSS3, Vanilla JavaScript;
- Серверная логика: отсутствует;
- Хранение состояния: `sessionStorage`;
- Источник предложений: локальный файл `banks.json`.

---

## 2. Общее описание системы

### 2.1 Страницы и назначение
- `index.html` — авторизация по номеру телефона и коду подтверждения;
- `banklist.html` — витрина предложений банков (загрузка из JSON);
- `application-form.html` — анкета заемщика с валидацией;
- `result.html` — профиль пользователя, новая заявка, мок-история и имитация результата.

### 2.2 Основной пользовательский поток
1. Пользователь открывает `index.html`;
2. Вводит телефон и подтверждает вход кодом;
3. Переходит на `banklist.html`;
4. Выбирает банк и переходит на `application-form.html`;
5. Отправляет анкету и переходит на `result.html`;
6. На `result.html` видит прогресс и итог новой заявки, может вернуться к списку банков.

### 2.3 Навигация между страницами
- `index.html` → `banklist.html` (при корректном коде);
- `banklist.html` → `application-form.html` (после выбора банка);
- `banklist.html` → `result.html` (кнопка «Профиль»);
- `banklist.html` → `index.html` (кнопка «Выход»);
- `application-form.html` → `banklist.html` (кнопка «К банкам»);
- `application-form.html` → `result.html` (после успешной отправки формы);
- `result.html` → `banklist.html` (кнопка «К списку банков»).

## 3. Функциональные требования

## 3.1 Модуль авторизации (`index.html`)

### FR-AUTH-01: Ввод телефона
Система должна принимать номер телефона в формате РФ:
- `+7XXXXXXXXXX`;
- `8XXXXXXXXXX`.

### FR-AUTH-02: Валидация телефона
Система должна отклонять пустой/некорректный номер и показывать сообщение об ошибке.

### FR-AUTH-03: Генерация и отправка кода (мок)
После корректного номера система должна:
- сгенерировать случайный 6-значный код;
- сохранить код во временное состояние страницы;
- показать код через `alert` (мок-отправка);
- вывести код в консоль для отладки.

### FR-AUTH-04: Подтверждение кода
Система должна проверять код как строку из 6 цифр. При совпадении — переход на `banklist.html`, иначе — ошибка.

### FR-AUTH-05: Повторная отправка
Система должна ограничивать повторную отправку кода таймером 60 секунд.

### FR-AUTH-06: Возврат к шагу телефона
Система должна позволять вернуться к вводу номера кнопкой «Изменить номер».

---

## 3.2 Модуль списка банков (`banklist.html`)

### FR-BANK-01: Загрузка предложений
Система должна загружать данные банков из файла `banks.json` через `fetch`.

### FR-BANK-02: Отображение карточки банка
Для каждого банка система должна отображать:
- `emoji` (или значение по умолчанию);
- `name`;
- `interestRate`;
- список `features` (или fallback-текст при отсутствии).

### FR-BANK-03: Выбор банка
При нажатии «Выбрать предложение» система должна:
- записать в `sessionStorage`:
  - `selectedBankName`;
  - `selectedBankRate`;
- перейти на `application-form.html`.

### FR-BANK-04: Состояния списка
Система должна поддерживать состояния:
- загрузка;
- ошибка (с кнопкой повтора);
- пустой список.

### FR-BANK-05: Доступ к профилю и выход
Система должна предоставлять:
- переход в профиль (`result.html`);
- выход на экран входа (`index.html`).

---

## 3.3 Модуль анкеты (`application-form.html`)

### FR-FORM-01: Разделы формы
Система должна содержать разделы:
1. Личные данные;
2. Паспортные данные;
3. Финансовые данные;
4. Дополнительно.

### FR-FORM-02: Обязательные поля
Система должна требовать заполнение полей:
- ФИО;
- дата рождения;
- телефон;
- серия/номер паспорта;
- кем выдан паспорт;
- дата выдачи паспорта;
- тип занятости;
- ежемесячный доход;
- сумма кредита;
- срок кредита;
- цель кредита;
- согласие на обработку данных.

### FR-FORM-03: Валидация значений
Система должна применять проверки:
- возраст: от 18 до 90 лет;
- телефон: формат РФ (`+7`/`8` + 10 цифр);
- email: опционален, но при заполнении проверяется формат;
- паспорт: 10 цифр (4+6);
- дата выдачи паспорта: не в будущем;
- доход: не менее 5000 ₽;
- сумма кредита: от 10 000 до 5 000 000 ₽;
- срок: выбор из списка (6/12/24/36/60 месяцев).

### FR-FORM-04: Обработка отправки
При успешной валидации система должна сохранять данные текущей заявки в `sessionStorage` и выполнять переход на `result.html`.

### FR-FORM-05: Очистка формы
Система должна очищать поля и ошибки по кнопке «Очистить».

---

## 3.4 Модуль профиля (`result.html`)

### FR-PROFILE-01: Отображение новой заявки
Система должна выводить новую заявку на основе данных из `sessionStorage`:
- ФИО пользователя;
- банк;
- сумма;
- срок;
- ставка;
- текущий статус.

### FR-PROFILE-02: Имитация обработки заявки
Для новой заявки система должна пошагово показывать статусы:
1. Проверка персональных данных;
2. Анализ кредитной истории;
3. Запрос решения банка.

После шагов система должна установить финальный статус случайным образом:
- `Одобрена`;
- `Отказана`.

### FR-PROFILE-03: Мок-история
Система должна отображать фиксированную (моковую) историю предыдущих заявок.

### FR-PROFILE-04: Возврат к списку банков
Система должна обеспечивать переход на `banklist.html` кнопкой «К списку банков».

---

## 4. Модель данных

## 4.1 Формат `banks.json`
Файл должен содержать массив объектов со структурой:
- `name: string`;
- `emoji: string`;
- `interestRate: number`;
- `features: string[]`.

## 4.2 Ключи `sessionStorage`
Система использует ключи:
- `selectedBankName`;
- `selectedBankRate`;
- `latestApplicationName`;
- `latestApplicationAmount`;
- `latestApplicationTerm`;
- `latestApplicationBank`;
- `latestApplicationRate`;
- `latestApplicationStatus`.

---

## 5. Нефункциональные требования

### NFR-01: Доступность
Интерфейс должен использовать ARIA-атрибуты и `aria-live` области для озвучивания статусов и ошибок.

### NFR-02: Адаптивность
Интерфейс должен корректно отображаться на мобильных и десктопных экранах (используются медиазапросы).

### NFR-03: UX-обратная связь
Система должна предоставлять визуальные состояния элементов управления:
- `hover`;
- `focus` / `focus-visible`;
- `active`;
- сообщения об ошибках.

### NFR-04: Ограничения реализации
- Отсутствует реальный backend;
- SMS-отправка не интегрирована (только mock);
- Решение по заявке в профиле не рассчитывается и выбирается случайно;
- Данные не сохраняются перманентно после завершения сессии браузера.

---

## 6. Критерии приемки (по текущей реализации)

1. Сквозной путь пользователя выполняется без серверной части:
   - `index.html` → `banklist.html` → `application-form.html` → `result.html`.
2. Выбор банка влияет на данные, отображаемые в анкете/профиле через `sessionStorage`.
3. Некорректные поля анкеты и авторизации блокируют успешное продолжение сценария.
4. Из `application-form.html` доступен возврат в `banklist.html`.
5. Из `result.html` доступен возврат в `banklist.html`.
6. Из `banklist.html` доступен переход в `result.html` и выход в `index.html`.

---

## 7. Запуск

Для корректной загрузки `banks.json` рекомендуется запуск через локальный HTTP-сервер.

Пример:

```bash
python3 -m http.server 8000
```

Открыть в браузере:
- `http://localhost:8000/index.html`
